# Hair Salon Simulator — Spec

## 1. コンセプト

**2つの柱** で構成される美容師シミュレーター：

### 柱① テクニック — リアルな髪物理演算によるヘアカット
ストランドレベルの物理演算で「1本1本切れる」リアルさ。
既存タイトル（Hairdresser Simulator等）のセクション分割式とは一線を画す。

### 柱② コミュニケーション — LLM会話による接客・謎解き
顧客とのリアルタイム会話（LLM駆動）で、ニーズを汲み取り、
ホスピタリティを発揮する。会話の質がリピート率・評判に直結する。

**ターゲット**: ゲーマー、美容師志望者、カジュアルシミュレーション好き
**差別化**: テクニック × 接客の2軸評価は市場に存在しない

---

## 2. ゲームデザイン概要

### 2.1 顧客来店フロー

```
来店 → カウンセリング会話 → 施術 → 仕上がり確認会話 → 評価 → 退店
         (柱②)           (柱①)      (柱②)         (①+②)
```

### 2.2 柱① テクニック評価

| 評価項目 | 内容 |
|---|---|
| カット精度 | 顧客の要望（長さ・スタイル）と仕上がりの一致度 |
| シンメトリー | 左右の対称性 |
| レイヤリング | 段の入れ方の自然さ |
| 仕上がり時間 | 速すぎず遅すぎない適切な時間 |

### 2.3 柱② コミュニケーション評価

| 評価項目 | 内容 |
|---|---|
| ニーズ把握 | 会話から本当の要望を引き出せたか（表面的な要望 ≠ 本音のケースあり） |
| 提案力 | 顧客に合ったスタイルを提案できたか |
| ホスピタリティ | 会話の心地よさ、気遣い、話題選び |
| 謎解き要素 | 特定の顧客は隠された悩み・ストーリーを持つ。会話で引き出すとイベント発生 |

### 2.4 リピート・評判システム

- 各顧客は **テクニック満足度** と **接客満足度** を個別に持つ
- 両方高い → リピート確定、口コミで新規顧客増加
- テクニックだけ高い → たまに来るが熱量低い
- 接客だけ高い → 「上手くないけど好き」→ リピートするが口コミは広がらない
- 両方低い → 二度と来ない、悪い口コミが広がる
- **評判レベル** が上がるとサロンのランクアップ、高単価顧客、難しい要望が解放

### 2.5 LLM会話システム設計

```
[顧客ペルソナ]        [会話エンジン]         [評価エンジン]
 性格・年齢・職業  →   LLM (Claude API)  →   発話分析
 髪の悩み・本音       System Prompt で      ニーズ抽出度
 隠しストーリー       ペルソナを注入         ホスピタリティスコア
                                            謎解き進捗
```

- **顧客ペルソナ**: JSON定義。性格（おしゃべり/寡黙/神経質等）、要望の明確度、隠しニーズ
- **LLM**: Claude API でリアルタイム会話。System Prompt にペルソナ + シーン状況を注入
- **評価**: 会話ログを別のLLM呼び出しで分析し、スコアリング
- **ヒント演出**: 顧客の表情変化（3Dモデル）や反応速度で、会話の方向性をプレイヤーにフィードバック

## 3. Tech Stack

| 項目 | 選定 | 理由 |
|---|---|---|
| Engine | Unity 6 (URP) | com.unity.demoteam.hair が公式サポート |
| Hair Physics | com.unity.demoteam.hair | GPU駆動、数万本ストランド、無料・公式 |
| Rendering | URP + カスタムヘアシェーダー | パフォーマンスとビジュアルのバランス |
| LLM | Claude API (Haiku / Sonnet) | 高品質な日本語会話、コスト効率 |
| LLM通信 | Unity C# → HTTPS (REST API) | SDK不要、シンプル |
| 会話UI | テキスト入力 + 選択肢ハイブリッド | 自由入力メイン、困った時の選択肢サポート |
| Input | Mouse + Keyboard（Phase 1） | VRは将来フェーズ |
| Platform | PC (Windows) | GPU性能が必要なためPC優先 |

## 4. Phase 1: ヘアカットデモ

### 4.1 ゴール

**「マネキンヘッドの髪をハサミで切れるデモ」** を作る。
まず柱①の物理演算＋カット操作の手触りを確認する。

### 4.2 必要な要素

#### A. マネキンヘッド
- 静的な頭部3Dモデル（アセットストアまたは無料モデル）
- 頭皮メッシュにヘアルートをマッピング

#### B. ヘアシステム
- com.unity.demoteam.hair をセットアップ
- ストランド数: 5,000〜10,000本（デモ用、パフォーマンス検証）
- 髪の長さ: ミディアム〜ロング（カットの変化が分かりやすい）
- 重力・風の基本物理
- ストランド同士の衝突（基本レベル）

#### C. ハサミツール
- マウスカーソル位置にハサミを表示
- クリック or ドラッグでカット操作
- カット判定: レイキャスト → ストランドとの交差検出 → 交差点でストランドを分割
- カットされた髪は短くなり、切り落とされた部分は落下（重力）

#### D. カメラ操作
- マネキンヘッドを中心にオービットカメラ
- ズームイン/アウト、右ドラッグで回転

#### E. UI（最小限）
- ツール選択（Phase 1ではハサミのみ）
- リセットボタン（髪を元に戻す）

## 7. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| com.unity.demoteam.hair がカット操作を想定していない | ストランド分割が困難 | ソースコードを読んで拡張。最悪、長さパラメータの動的変更で代替 |
| GPU負荷が高すぎる | フレームレート低下 | ストランド数削減、LOD活用、ガイドヘア方式で間引き |
| 頭部モデルとヘアルートの整合性 | 髪が頭を貫通 | SDFベースの衝突で対処 |
| LLMレスポンス遅延 | 会話テンポが悪い | Haiku使用でレイテンシ最小化、ストリーミング表示 |
| LLM APIコスト | 運用コスト増 | Haiku（安価）メイン、評価のみSonnet。ローカルLLMも将来検討 |
| 会話の品質ブレ | 不自然な応答 | ペルソナPromptの作り込み、ガードレール設定 |
| 評価の公平性 | スコアが恣意的に感じる | 評価基準をプレイヤーに明示、複数回LLM評価で平均化 |
